## 기술 설계

### 데이터 흐름

```
[클라이언트]
    │
    ▼ GET /api/follows/feed
[BFF route.ts]
    │
    ├─ auth check (session 없으면 401)
    ├─ getFollowedArtistIds(userId)  → artist_ids[]
    ├─ 각 artist_id → getPerformancesByArtist(artistId) → mt20id[]
    ├─ mt20ids 합집합(Set) — 중복 제거
    │
    └─ KOPIS pblprfr 호출(오늘~오늘+90일, 전국, 31일 분할 3회)
        │  기존 mergePerformances + mapWithConcurrency 재사용
        │  D3(31일 제한) 우회: 3개 기간으로 분할 병렬 호출
        ▼
       결과를 mt20id Set으로 교차 필터링
        ▼
       prfstate != '공연완료' 필터(공연예정·공연중만)
        ▼
       prfpdfrom ASC 정렬
        ▼
    { performances: PerformanceSummary[] }
```

### 31일 분할 전략

KOPIS API는 최대 31일 범위를 허용한다(D3). 90일을 커버하려면 3개 기간으로 분할:
- 기간 1: today ~ today+30일
- 기간 2: today+31 ~ today+60일
- 기간 3: today+61 ~ today+90일

각 기간당 최대 2페이지(60건) 호출 → 총 최대 180건 → mt20id 교차 필터.

### 빈 상태 처리

- 팔로잉 아티스트가 없는 경우 → "아직 팔로잉한 아티스트가 없어요" + `/following` 링크
- 팔로잉 아티스트는 있으나 다가오는 공연이 없는 경우 → "팔로잉 아티스트의 다가오는 공연이 없습니다"
- 둘 다 빈 상태를 구분해서 안내
