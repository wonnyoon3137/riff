## D1~D10 충돌 / 리스크 검토 결과

docs와 20개 이슈를 대조한 결과 **D 결정과 정면 충돌하는 이슈는 없다.** 단, 진행 전 처리해야 할 게이트 4건:

- **R1 (게이트) — F-1(#8) 검색은 D10 갱신 선행 필수.**
  D10 = "검색 v0.1 제외", features.md F5 = "v0.2 이관". 이슈 본문도 "features.md D10 갱신 선행"을 명시. KOPIS `shprfnm` 파라미터는 검증됨(kopis-integration §2.1). **코드 착수 전 `features.md`의 D10/F5를 먼저 갱신**해야 한다(결정 변경은 문서 먼저 원칙). 갱신 없이 검색 구현 시작 = 범위 위반.

- **R2 — F-2(#9)는 T-01·T-02·T-04에 의존.** D5 동기화 DB 구현(prfplc 전량 fetch)은 KOPIS rate limit(T-01)과 라벨→코드 역매핑(T-04)이 선결되어야 안전하다. F-2를 T-01/T-04보다 먼저 착수하면 동기화 중 rate limit에 막히거나 sidoCode/gugunCode가 비는 불완전 DB가 된다.

- **R3 — F-3(#10) ↔ T-08(#19)는 사실상 동일 작업.** 둘 다 `PeriodFilter.tsx:74`의 `alert("최대 31일까지 선택 가능합니다")` 교체. screens.md §139·§186에 토스트 스펙(하단중앙/좌하단, 3초, inverse-surface, 문구)이 이미 확정돼 있어 신규 결정 불필요. **하나의 작업으로 합쳐 처리**한다. 토스트 도입 후 `no-alert` 린트 유지(P-2/#2 연계).

- **R4 — T-03(#14) 구군 데이터화는 D4 구군 드릴다운의 전제.** kopis-codes §4.2가 "구군 표는 PDF/행정표준코드에서 생성"으로 미완 표기. 구군 단위 필터(features F2.2 2단계)를 v2에 넣으려면 선결. 단 kopis-codes §4 주석대로 "시도까지만 v0.1, 구군 v0.2 축소"가 가능하므로 **구군 범위 포함 여부는 제품 결정**(아래 R5).

- **R5 (제품 결정 요청) — 구군 드릴다운을 v2 범위에 포함할지.** 포함 시 T-03(#14) 데이터화 + T-04(#15) 역매핑이 선결로 들어온다. planner는 추측하지 않고 사람 결정을 요청한다.

#### 측정 노트 (2026-06-12, backend-dev)

**#12 T-01 — rate limit/가동률 실측: 완료 (2026-06-12, 키 주입 후 실측)**
- 하네스 `scripts/measure/rate-limit-probe.ts` 를 실 서비스키(`.env`)로 실행. 단계별 30req, rows=10, timeout=5s.

| 동시성 | 성공률 | 처리량(req/s) | 지연 p50/p95/max(ms) | 비고 |
|---|---|---|---|---|
| 1 | 100% | 6.2 | 157 / 203 / 320 | 안정 |
| 2 | 100% | 10.7 | 186 / 246 / 302 | 안정 |
| **5** | **100%** | **11.2** | 361 / 820 / 827 | 처리량 정점, 지연 증가 시작 → **안전 상한** |
| 10 | 100% | 10.0 | 798 / 1946 / 3002 | 지연 급증(max≈timeout), 처리량 정체 |
| 20 | **63.3%** | 15.0 | 484 / 1867 / 1990 | **HTTP 400 11건 — 포화/거부 시작** |

- **결론:** 안전 동시성 상한 = **5**(100% 성공 + 처리량 정점, 지연 수용 가능). 10은 지연만 악화되고 처리량 이득 없음. 20에서 명확히 포화(성공률 63%). → 다중지역 병합 batch 동시성 기본값 5(§7.2)는 실측과 일치. `KOPIS_REGION_CONCURRENCY` 로 조정 가능하나 5 권장.
- **F-2(공연장 전량 동기화) 함의:** prfplc 대량 fetch도 동시성 ≤5로 batch + 재시도. 동기화 주기(T-05/#16)는 이 처리량(≈11 req/s @ c5)으로 산정.
- ⏳ **후속(권한 필요):** 이 수치를 `kopis-integration` 스킬 §호출전략/§에러·캐싱("rate limit 미검증→실측")에 반영해야 함. 스킬 문서는 에이전트 설정이라 자동수정 차단됨 → 사람 승인 후 갱신.

**#13 T-02 — 지역 다중 병합 페이지네이션 정확도/부하: 검증 완료, 결함 검출**
- 범위(R5): **시도 레벨 병합만** 검증. 구군은 v0.2 보류 → 대상 아님.
- 검증 도구(서비스키 불필요, 결정적 합성 데이터):
  - 하네스 `scripts/measure/region-merge-verify.ts` (`pnpm tsx ...`, 결함 시 exit 2).
  - 회귀 테스트 `src/__tests__/region-merge-pagination.test.ts` (vitest, 그린).
- **정확도 결과(결함):** 현행 `src/app/api/performances/route.ts` 의 다중지역 경로는 각 지역을 `cpage=page` 로 호출하고 병합 후 항상 `slicePage(merged, 1, rows)` 한다. 지역별 페이지 경계가 전역 정렬 병합 경계와 어긋나 **클라이언트 page>1 에서 누락 발생**(검증 시나리오: 지역 3개×80건, rows=30, page 2~4에서 각 30건 누락 = 누락 90건). 즉 무한스크롤 다음 페이지에서 항목이 통째로 빠진다(R5 시도 병합에서도 재현).
- **부하 결과:** 호출 수 = 페이지당 지역 수 N (O(N)). 시도 최대 17개 → 페이지당 ≤17 병렬 호출. §7.2 동시성 상한(기본 5)으로 batch 분할 필요. 실 지연/거부 임계는 #12 키 확보 후 실측으로 보정.
- **수정안(F-2 착수 전 적용 권장):** 다중지역은 각 지역을 `cpage=1..page`(또는 `page*rows` 충분량) 누적 fetch → 전역 병합·dedup·정렬 → `slicePage(merged, page, rows)` 로 전역 오프셋 슬라이스. 제안 알고리즘은 모든 페이지에서 ground-truth 와 일치함을 하네스·테스트로 확인.
- ✅ **수정 완료(2026-06-12, #21):** 위 수정안 적용. `src/server/kopis/concurrency.ts`(`mapWithConcurrency`, 상한 5) 신규 + `route.ts` 다중지역 경로 재작성. `region-merge-verify.ts` exit 0(전 페이지 누락 0/중복 0), `region-merge-pagination.test.ts` page>1 누락 0 단언 그린. vitest 38 pass.
