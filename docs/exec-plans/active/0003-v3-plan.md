## 트랙 P1 — ① 탐색 강화 (의존 0, 기존 토대 재사용)

> 기존 자산 재사용(확인됨): `src/domain/filter-url.ts`(URL 동기화·filterHash), `src/domain/types.ts`(FilterState), `src/domain/kopis-codes.ts`(시도표·시도 역매핑), `src/server/kopis/merge.ts`+`concurrency.ts`(D4 병합·#21 누락 수정·동시성 5), `src/app/api/performances/`(BFF), `src/components/filters/`(PeriodFilter·RegionFilter·GenreFilter·VenueFilter).

### P1-1. 가격대 필터 (`pcseguidance` 후처리)

**KOPIS 가능/불가능 (정직히):**
- KOPIS `/pblprfr`에는 **가격 검색 파라미터가 없다**(skill §18 선택 파라미터 목록: shcate/signgucode/signgucodesub/prfplccd/prfstate/openrun — 가격 없음). 가격은 **상세 응답의 `pcseguidance` 자유 텍스트**로만 존재(예 `"R석 99,000원 / S석 88,000원"`, data-model §3.2).
- ∴ **가격대 필터는 KOPIS 검색으로 불가능. BFF/클라 측 후처리(파싱→밴드→필터링)** 로만 구현 가능. 이는 0002 검색(#8, shprfnm 위임)과 구조가 **근본적으로 다르다**(서버 위임 불가).
- **추가 제약:** `pcseguidance`는 **목록(pblprfr 목록) 응답에 없고 상세 응답에만 있다**(data-model §4.1엔 priceGuidance 없음, §4.2에만 존재). 목록 카드를 가격으로 거르려면 **상세를 추가 호출**해야 함 → 부하·구조 문제. **이 점이 P1-1의 핵심 설계 결정**(아래 DEC-P1).

**파싱 → 가격 밴드 (보수적):**
- 밴드 예: `무료/초대` · `~3만` · `3~7만` · `7만+` · `가변·미정`(파싱 실패/전석 초대/공연별 상이).
- 엣지: `"전석 초대"`·`"무료"` → 무료/초대. `"전석 5만원"` → 단일가. `"균일가"`. 좌석등급 다수(`"VIP 12만/R 9.9만/S 8.8만"`) → **최저가 기준 밴드**(보수적). `"가격 문의"`·빈 문자열·숫자 없음 → 가변·미정. 통화·천단위 콤마·"원"·"won" 표기 정규화.
- **100% 정확 불가 전제.** 파싱 실패는 버리지 않고 `가변·미정` 밴드로 포함(거짓 제외 방지). vision §5.1·V-4 권고.

**DEC-P1 (설계 결정 — backend-dev 주도, planner 추천):** `pcseguidance`가 상세에만 있으므로 가격대 필터를 어디서 적용?

| 옵션 | 동작 | 평가 |
|---|---|---|
| A. 상세 N회 호출 후 필터 | 목록 각 항목의 상세를 호출해 가격 추출 후 거름 | 부하 폭발(페이지당 30×상세호출), rate limit 위협. **비권장** |
| B. Artist DB처럼 가격을 사전 동기화 | Venue 선례로 공연 가격을 자체 DB에 적재 후 필터 | 비저장 원칙 큰 완화·동기화 파이프라인 필요. v3 P1엔 과함 |
| **C. 상세 진입 시점 추출 + 표시(밴드 칩), 필터는 보류/축소** | 상세 페이지에서 `pcseguidance`→밴드 칩 표시. 목록 필터는 "데이터 가용성 확인 후" | **planner 추천(보수적).** 가격이 목록에 없다는 KOPIS 제약을 정직히 반영 |

> **planner 권고:** 가격대 **필터**(목록 거르기)는 KOPIS 구조상 비싸다(상세 N회). v3에서는 **(1) 상세 페이지 가격 밴드 표시(저비용·즉시 가치)** 를 먼저 하고, **(2) 목록 필터는 V-4(밴드 정밀도)+가용성 실측 후** 옵션 B(사전 동기화)와 묶어 결정하길 권한다. 이 분리 여부는 **사람 결정(아래 결정 포인트 V-4 연계)**.

**담당/선행/QA:**
- 선행 docs: `features.md` F2에 가격대 필터 항목 추가(필터 vs 표시 범위 명시) — 코드 착수 전.

### P1-3. 월별 필터 (기존 기간 필터와 중복 검토 — 범위 판정)

**현황:** 기간 필터(D1·D3, `FilterState.period`)는 이미 `preset: THIS_MONTH | NEXT_MONTH | MONTH_AFTER | CUSTOM | DEFAULT_30D` + from/to 범위를 가짐(data-model §3.4). features F2.1 "이번 달/다음 달/그 다음 달/직접 선택"이 **이미 월 단위 빠른 선택을 포함**한다.

**판정(planner):** "월별 필터"는 **이미 기존 기간 필터(F2.1)에 구현·정의되어 있다.** 별도 신규 UI를 추가하면 중복. ∴ **P1-3은 신규 구현 범위에서 제외**하고, 다음만 정리:
- (정합성 점검) F2.1 프리셋(THIS/NEXT/MONTH_AFTER)이 31일 제약(D3) 안에서 정확히 월 경계로 매핑되는지 qa 확인(예: 1월=31일 OK, 2월 등). 이미 구현됐다면 회귀 확인만.

**담당/QA:** 신규 작업 없음. **QA 티어 = Fast**(기존 프리셋 회귀 점검). 신규 UI 도입 시에만 재계획.

### P2-① prfcast 자유텍스트에서 출연진 추출 (난이도: 높음)

- 입력: 상세 `prfcast`(자유 텍스트, 원문 보존). 예 `"홍길동, 김철수 외"`, `"음악감독 ○○○ / 출연 △△△, ◇◇◇"`.
- 작업: 사람/팀 이름 토큰 분리·정제(구분자 `,`·`/`·`외`·역할어 제거). NER 또는 규칙 기반.
- 리스크: 배역·역할·구분자 혼재, 노이즈 큼(vision §3.1). **정확도 100% 불가** — 후단 수동 보정(P2-③) 전제로 설계.
- 원문 보존 불변(G-3): 추출은 별도 테이블에 사본 적재, `prfcast` 원문은 UI에 그대로 유지.
- 담당: backend-dev. QA 티어=**Deep**(추출 정밀도/재현율 측정).

### P2-② 외부 음악 API 매칭 (난이도: 중~높, V-8 결정 연계)

- 작업: 추출된 이름을 외부 음악 메타 API로 조회 → 아티스트 식별·정규화·메타 보강(표기 변형 해소: "방탄소년단"="BTS").
- **외부 API 선택 = V-8(외부 API 정책)와 연계 — 결정 포인트로 올림(아래).** 후보 비교는 결정 포인트 섹션 참조.
- 리스크: API 약관·쿼터(KOPIS rate limit 실측 선례처럼 **실측 필요**), 매칭 정확도, **클래식/연극/국악 등 비대중음악 커버리지 한계**(vision §3.1). 대중음악 외 장르는 매칭 실패율 높을 것 → 수동 보정(P2-③)으로 흡수.
- 담당: backend-dev. QA 티어=**Deep**(매칭 정확도·**오매칭률** 측정 — 아래 QA 기준).

### P2-③ 수동 보정 (난이도: 중, 품질 게이트)

- 매칭 실패/모호 케이스를 사람이 교정(아티스트 마스터 시드·연결 승인). 운영 도구는 v3에선 최소(스크립트/시드 파일 수준, 풀 어드민 UI는 후속).
- 리스크: 확장성(인건비). 인기 아티스트 우선 시드(vision §3.1 (d) 하이브리드 권고).
- 담당: backend-dev(데이터 도구) + 사람(큐레이션). QA: 보정 반영 멱등성.

### P2-④ 자체 DB 저장 (난이도: 중, Venue 선례 재사용)

- `artists` 테이블 + `performance_artists` 관계 테이블(SQLite, better-sqlite3 — DEC-A 선례). 외부 API ID·표기변형·매칭 신뢰도 컬럼. upsert 멱등(Venue sync 선례).
- 비저장 원칙 완화(V-9 부분): Artist는 영속(Venue처럼 예외). **공연(Performance)은 여전히 비저장** — 관계 테이블은 `mt20id`(KOPIS 키)를 외래 참조만.
- 담당: backend-dev. QA 티어=**Deep**(스키마·upsert 멱등·관계 무결성).

### P2 인증 불필요 / 비범위

- P2는 **순수 데이터 토대** — 사용자 계정 불필요(V-2 미결정과 무관하게 진행 가능).
- **②Following(팔로우)은 P2 이후 별도 phase**(P5, 인증=V-2 전제). v3 P2의 사용자 가시 산출물은 **상세 페이지의 출연 아티스트 칩(클릭=동일 아티스트 다른 공연, 단 P3 필터는 별도)** 수준까지만(vision §5.1).

### P2 QA — 매칭 정확도/오매칭률 측정 기준

- **정밀도(precision):** 매칭된 (이름→아티스트) 쌍 중 올바른 비율. 오매칭률 = 1−정밀도.
- **재현율(recall):** 실제 추출 가능한 출연자 중 매칭 성공 비율.
- 측정: 장르별 샘플(대중음악 vs 클래식/연극) 수동 정답셋 N건 대비. **장르별 분리 보고**(비대중음악 커버리지 한계 정량화).
- 게이트(권고): 대중음악 정밀도 우선, 비대중음악은 수동 보정 비중 명시. 임계값은 첫 측정 후 사람과 확정.
