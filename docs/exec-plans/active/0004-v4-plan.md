## 목표

v4는 **두 축**으로 구성한다:

1. **품질 보강 (Quality Gate)** — v3 회고 부채(R3/R4/R5) 상환. P2 데이터 토대의 실측 검증 없이 그 위에 P3을 쌓으면 모래 위의 성이 된다. **품질 보강을 선행 게이트로 두고, P3 착수 전에 완료**한다.
2. **P3 아티스트별 필터** — vision.md P2(Artist 엔티티) 위에서, 아티스트로 공연을 필터링하는 기능. 인증(P4) 불필요. P2 데이터가 실측으로 검증된 후에만 착수.

> **비범위(v4 밖):** 인증(P4, V-2 미결정), Following(P5), My Record(P6), 디깅(P9), 가격대 **목록 필터**(사전 동기화 필요, 후속). P3는 "상세 칩 클릭 이동"(F7.2, v3 구현)을 **목록 페이지 필터 UI로 승격**하는 것이다.

### V-10 planner 추천 근거

- **P3(아티스트별 필터)는 P2 데이터 위에서 인증 없이 구현 가능**(vision S4.1 의존 그래프: P2->P3, 인증 불필요). v3에서 P2 토대를 깔았으므로 자연스러운 다음 단계.
- **P4(인증)는 큰 제품 전환**(V-2 미결정, 약관/개인정보/세션). P3보다 의존이 깊고, 사용자 가치 직결이 아님(인프라). P3을 먼저 출하해 ① 탐색 도메인을 완성하는 것이 ROI가 높다.
- **품질 보강만(나)은 안전하나 기능 진전 없음.** R4(MB 실측)가 P3의 품질 게이트이므로 (가)에서 자연스럽게 선행-후행으로 묶인다.
- 단, **R4 실측 결과가 나쁘면**(매칭률 극히 낮음) P3 착수를 재검토해야 한다. 이 경우 P2 데이터 보강(외부 API 추가, 수동 시드 확대)이 우선이며 P3은 보류.

> 이것은 planner 추천이다. 최종 결정은 사람(혜원)이 한다.

## 트랙 P3 — 아티스트별 필터 (P2 데이터 위, 인증 불필요)

> **선행 게이트: Q-2(MB 실측) 통과 + G-4/G-5(docs 갱신) 완료.**
> 현재 F7.2(상세 페이지 아티스트 칩 클릭 -> 해당 아티스트 공연 이동)를 **목록 페이지의 1급 필터로 승격**한다.

### 설계 판단 (planner)

**KOPIS 가능/불가능 (정직히):**
- KOPIS `/pblprfr`에는 **아티스트/출연진 검색 파라미터가 없다**(skill S18 선택 파라미터: shcate/signgucode/signgucodesub/prfplccd/prfstate/openrun/shprfnm). 출연진은 상세 응답의 `prfcast` 자유 텍스트에만 존재.
- 따라서 아티스트별 필터는 **KOPIS 위임 불가**. 자체 Artist DB(`performance_artists` 관계 테이블)에서 `mt20id` 목록을 조회한 뒤 **BFF에서 필터링**하는 구조.

**구현 전략:**
1. 자체 Artist DB에서 `artistId` -> `mt20id[]` (해당 아티스트 출연 공연 ID 목록) 조회
2. 이 `mt20id[]`로 KOPIS 상세를 조회하거나, 기존 목록 결과와 교차(intersection)
3. 기존 필터(기간/지역/장르/공연장/검색)와 AND 결합

**제약:**
- Artist DB에 적재된 공연만 필터 가능 (파이프라인 미처리 공연은 누락)
- 파이프라인 커버리지가 필터 품질을 결정 -> Q-2 실측이 선행 게이트인 이유
