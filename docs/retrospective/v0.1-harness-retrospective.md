## 1. 하네스 구조 요약

Riff는 **에이전트 기반 하네스 엔지니어링**으로 개발했다. OpenAI의 "AGENTS.md를 100줄 목차로, 상세는 docs/"와 revfactory의 ".claude/agents + skills 파일화"를 조합한 구조다.

### 1.1 구성 요소

```
AGENTS.md (루트 목차, ~73줄)
├── docs/ (진실원본 — product, architecture, api, exec-plans)
├── .claude/agents/ (역할별 서브에이전트)
│   ├── planner.md      — 작업 분해, D1~D10 정합성 검토
│   ├── frontend-dev.md — S1/S2 화면, M3 컴포넌트, 상태 관리
│   ├── backend-dev.md  — BFF, KOPIS 연동, 정규화, 병합
│   └── qa-engineer.md  — 검증 게이트, 경계면 교차 비교
└── .claude/skills/ (도메인 지식 캡슐화)
    ├── orchestrator/   — 전체 조율, docs-우선, 완료 기준
    ├── nextjs-patterns/ — App Router, React Query, M3 패턴
    └── kopis-integration/ — KOPIS API 연동 실무 규칙
```

### 1.2 핵심 원칙

1. **docs-우선**: 코드보다 문서를 먼저 읽고, 결정 변경 시 docs를 먼저 갱신
2. **D1~D10 게이트**: 10개 불변 의사결정이 모든 구현의 가드레일
3. **생성-검증 파이프라인**: planner → dev → qa-engineer, 실패 시 planner 환류
4. **피드백 루프**: `pnpm typecheck && pnpm lint && pnpm test` 그린 = 완료

## 2. 실제 실행 과정

### Phase A+B — 하네스 미적용 (교훈 발생)

| 항목 | 내용 |
|---|---|
| 커밋 | `5c28c90` |
| 방식 | orchestrator/agent 위임 없이 직접 구현 |
| 산출물 | Next.js 스캐폴딩, 도메인 타입, KOPIS BFF, 테스트 33개 |

**문제점**: AGENTS.md에 하네스 패턴(orchestrator → agent 위임)을 정의해 놓고, 실제로는 에이전트 위임 없이 단일 흐름으로 구현했다. 사용자가 이를 지적:

> "AGENTS.md에 정의된 하네스 패턴(orchestrator가 조율 → 에이전트 위임 → 스킬 적용)을 제대로 따라야지 !!!!!!!!"

**교훈**: 하네스 구조를 설계만 해놓고 실행에서 생략하면, 하네스의 가치(역할 분리, 검증 게이트, 문서 정합성)가 사라진다. Phase A+B 결과물 자체는 정상이었지만, 프로세스 준수가 보장되지 않았다.

## 4. 무엇이 잘 되었나

### 4.1 docs-우선 원칙의 효과
- 8개 docs 문서가 전체 구현의 진실원본 역할을 했다
- D1~D10 게이트 덕분에 범위 이탈이 없었다 (검색/지도/로그인 등 v0.1 외 기능 0건)
- kopis-codes.md의 "verified" 마킹으로 미검증 코드값 하드코딩을 사전 차단

### 4.2 planner → dev → qa 파이프라인
- planner의 28개 작업 분해가 구현 순서와 병렬성을 명확히 했다
- qa-engineer의 18개 항목 전수 검사가 누락을 잡아냈다
- 경계면 교차 비교(BFF 응답 ↔ 훅 ↔ 컴포넌트)로 타입 불일치 0건

### 4.3 실행계획의 일급 산출물화
- `docs/exec-plans/active/0001-project-bootstrap.md`가 "살아있는 문서"로 작동
- 진행 로그와 의사결정 로그가 누적되어 맥락 손실 없음
- 체크포인트(Phase C 전 사람 승인)가 정상 작동

### 4.4 _workspace/ 패턴
- 중간 산출물(planner 작업분해, qa 검증보고)을 `_workspace/`에 분리
- gitignore 처리로 최종 커밋에는 불필요한 파일 미포함
- 단, 검증보고서의 가치가 높아 docs/로 승격할지 검토 필요

## 5. 무엇이 아쉬웠나

### 5.1 Phase A+B에서 하네스 미적용
- **원인**: 하네스 구조를 설계한 직후라 "작은 작업은 직접 해도 될 것"이라는 판단
- **결과**: 프로세스 일관성 깨짐, 사용자 신뢰 저하
- **개선**: v0.2에서는 작업 규모와 무관하게 orchestrator → agent 파이프라인을 일관 적용

### 5.2 커밋 단위의 아쉬움
- Phase A+B가 하나의 거대 커밋(`5c28c90`)으로 묶임
- Phase C도 2개 커밋으로 압축(이상적으로는 그룹별 커밋)
- **개선**: v0.2에서는 논리 단위별 커밋 + 커밋 메시지에 Phase/Group 태그

### 5.3 E2E 테스트 부재
- 단위 테스트 33개는 모두 서버/도메인 레이어
- 핵심 플로우(필터 → 목록 → 상세 → 뒤로가기 복원)의 E2E 미작성
- **개선**: Phase D에서 Playwright 기반 E2E 추가 필요

### 5.4 D3 보정 피드백 UX
- 31일 초과 시 `alert()` 사용 — 접근성/UX 열악
- **개선**: 인라인 토스트/스낵바 컴포넌트로 교체

### 5.5 에이전트 간 실시간 소통 부재
- planner → frontend-dev 전달이 TaskCreate/파일 기반이라 피드백 루프가 느림
- **개선**: v0.2에서 더 세밀한 태스크 단위와 중간 체크포인트 도입 검토
