## 1. v0.2 → v0.3 무엇이 달라졌나

v0.2는 "반복 가능한 개발 사이클"의 증명이었다. v0.3는 **"새로운 도메인(Artist)을 docs-우선으로 도입하고, 기존 토대를 확장(구군·가격)하는 첫 기능 확장 사이클"** 이다.

### 1.1 새로 도입한 것

- **제품 비전 문서** (`docs/product/vision.md`): 4도메인(탐색·팔로우·디깅·기록) 정의, 단계적 로드맵(P1~P9), 사람 결정 항목(V-1~V-9) 분리.
- **Artist 1급 엔티티**: KOPIS에 없는 개념을 하이브리드(추출+외부API+수동보정)로 자체 구축. 비저장 원칙의 두 번째 예외(Venue 선례).
- **외부 API 연동**: MusicBrainz API PoC — 프로젝트 최초의 KOPIS 외 외부 API.
- **운영 스크립트**: artist-seed/verify/run-pipeline — 데이터 파이프라인 CLI 도구.

## 2. 잘한 것

### 2.1 docs 게이트 준수
G-1/G-2/G-3을 코드 착수 전에 충족시켰다. 특히 G-3(kopis 스킬 prfcast 파싱 예외)은 "원문 보존 불변 + 별도 사본 적재"라는 경계를 명확히 해서, 기존 원칙을 깨지 않으면서 새 기능을 도입할 수 있었다.

### 2.2 병렬 에이전트 활용
P1-1/P1-2/P2-① 3건, P1-2 UI/P2-②/P2-④ 3건을 각각 병렬로 실행해 처리 속도를 높였다. 독립적인 작업을 잘 분리한 계획서(0003)가 병렬화를 가능하게 했다.

Artist DB(repo.ts)를 Venue repo 패턴 그대로 재사용해 설계 시간을 줄이고 일관성을 유지했다. better-sqlite3, upsert 멱등, DI 패턴 등.

### 2.4 테스트 커버리지 대폭 확장
153개 신규 테스트. 특히 가격 밴드(34), 구군 매핑(54), 파이프라인(13)은 엣지케이스를 충분히 커버했다.

## 3. 못한 것 / 교훈

### 3.1 [Critical] 프로세스 위반 — 이슈/브랜치 없이 main 직접 작업
- **사실**: v3 전체를 main에서 직접 개발하고, 이슈 없이 커밋했다. PR #38은 사후에 브랜치를 분리하고 이슈를 소급 생성했다.
- **원인**: v2 회고에서 정립한 워크플로(이슈→브랜치→PR)를 인지하고 있었으나, "빨리 코드를 짜야 한다"는 관성에 밀려 생략했다.
- **영향**: 이슈 기반 추적 불가, 코드 리뷰 없이 main 오염 위험, PR이 너무 커짐(38파일, +4402줄).
- **교훈(R1)**: **이슈 먼저, 브랜치 분리는 "자동 반사"로 해야 한다.** 어떤 작업이든 첫 행동은 `gh issue create` + `git checkout -b`. 코드부터 치지 않는다.

### 3.2 PR 1개에 모든 것을 담음
- **사실**: P1-1, P1-2, P2 전체를 하나의 PR #38에 넣었다. 38파일 변경, +4402줄.
- **원인**: 이슈/브랜치 분리를 안 해서 자연스럽게 단일 PR이 됨.
- **교훈(R2)**: **PR 단위 = 이슈 단위.** 최소 P1-1, P1-2, P2를 각각 별도 PR로 분리했어야 한다. 리뷰 가능한 크기(~500줄)를 목표로.

### 3.3 E2E 테스트 미작성
- **사실**: 단위 테스트는 297개 작성했으나, 구군 드릴다운이나 가격 밴드 칩 등의 E2E 테스트는 작성하지 않았다.
- **교훈(R3)**: v0.2에서 E2E를 P0으로 도입한 선례가 있다. 새 UI(RegionFilter 드릴다운, PriceBandChip)에 대한 E2E를 후속으로 작성해야 한다.

### 3.4 MusicBrainz 실측 미수행
- **사실**: MB API PoC를 mock 테스트로만 검증했다. 실제 한국 아티스트 매칭률, rate limit 체감, 약관 세부는 미실측.
- **교훈(R4)**: KOPIS rate limit을 실측한 v0.1 선례처럼, **MB API도 실 데이터로 매칭률·응답시간을 실측**해야 한다. PoC 결과 없이 P2-②를 "완료"로 표기하는 것은 과장.

### 3.5 구군 코드표 KOPIS 실 데이터 미검증
- **사실**: 행정표준코드 기반으로 228개 구군을 데이터화했으나, KOPIS가 실제로 이 코드를 수용하는지 실 호출로 검증하지 않았다.
- **교훈(R5)**: 구군 코드 = 행정표준코드와 KOPIS 코드가 **동일하다는 가정**. 실 호출로 최소 서울/경기 주요 구군을 검증해야 한다.
