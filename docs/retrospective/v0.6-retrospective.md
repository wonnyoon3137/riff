## 1. v0.5 → v0.6 무엇이 달라졌나

v0.5는 "인증 토대 구축(OAuth, DB, 로그인 UI)"이었다. v0.6은 **"인증 위에 첫 개인화 기능을 올린 사이클 — 아티스트 팔로우/언팔로우 + Following 목록 페이지"**이다.

### 1.1 새로 도입한 것

- **F-1 docs 정비**: features.md F10 Following 정의, screens.md S4 Following 목록 + ArtistChip 팔로우 버튼 상태, data-model.md follows 엔티티(DDL/쿼리 패턴).
- **P5-1 follows DB + BFF API**: users.db에 follows 테이블(user_id × artist_id 복합 PK) 추가. `GET/POST/DELETE /api/follows` Route Handler. 단위 테스트 8건.
- **P5-2 Follow UI**: ArtistChip 팔로우/팔로잉 토글 버튼(optimistic update + 실패 롤백). `/following` 목록 페이지(미들웨어 인증 보호). AppBar 드롭다운 "팔로잉 목록" 링크.
- **P5-3 Following E2E**: 5개 시나리오(비로그인 버튼 미표시, 로그인 버튼 표시, 팔로우 토글, /following 리디렉션, /following 목록 렌더링). `e2e/fixtures/auth.ts`(JWE 쿠키 헬퍼) 신규 추가.

### 2.2 Auth.js 미들웨어 인증 레이어 구분 발견

Playwright E2E에서 두 인증 레이어를 구분해야 했다.

- **미들웨어 레이어**: `authorized` 콜백이 실제 JWE 쿠키를 검증 → `/api/auth/session` 모킹만으로는 보호 경로 진입 불가
- **클라이언트 레이어**: `useSession()`이 `/api/auth/session`을 호출 → 세션 응답 모킹 필요

두 레이어 모두 처리하기 위해 `@auth/core/jwt`의 `encode`를 사용한 `setAuthCookie()` 헬퍼(`e2e/fixtures/auth.ts`)를 작성했다. 이 패턴은 향후 인증이 필요한 모든 E2E 테스트에서 재사용된다.

### 2.3 Playwright 라우트 우선순위 발견

Playwright는 나중에 등록된 라우트가 먼저 매칭된다. 일반 BFF mock(`mockPerformanceApi`)을 먼저, 특정 경로 mock(`mockFollowablePerformanceDetail`)을 나중에 등록해야 특수 픽스처가 우선 적용된다. 첫 구현에서 순서를 반대로 하여 타이틀 불일치 오류가 발생했다.

### 2.4 낙관적 업데이트 E2E 검증 패턴 구축

`onSettled`의 `invalidateQueries` 재조회가 mock 상태를 덮어써 팔로우 버튼이 원래 상태로 돌아가는 문제가 있었다. `let followed = false` 상태 변수를 가진 stateful route mock으로 해결했다. 낙관적 업데이트 + 서버 재조회 흐름을 E2E에서 완전히 검증하는 패턴을 확립했다.

### 2.5 병렬 E2E worker 수 조정

ArtistChip의 `"use client"` 전환 이후 상세 페이지 진입 시 `useSession` 호출이 추가되어 dev 서버 부하가 증가했다. `workers: undefined`(CPU 수)에서 `workers: 2`로 줄여 artist-filter 테스트의 간헐적 실패를 해소했다.

## 3. 못한 것 / 교훈

### 3.1 `@auth/core/jwt` 직접 import로 typecheck 실패

- **사실**: `e2e/fixtures/auth.ts`에서 `@auth/core/jwt`를 직접 import했으나, pnpm 유령 의존성 보호로 인해 `@auth/core`가 `node_modules`에 없어 typecheck가 실패했다.
- **원인**: pnpm은 직접 의존성이 아닌 패키지를 `node_modules` 루트에 노출하지 않는다. `@auth/core`는 `next-auth`의 간접 의존성.
- **수정**: `next-auth/jwt`로 import 경로 변경(`next-auth`가 `@auth/core/jwt`를 re-export).
- **교훈(R17)**: **pnpm 프로젝트에서 간접 의존성을 직접 import하지 않는다.** 직접 의존성(`next-auth`)의 re-export를 사용하거나 `package.json`에 명시적으로 추가한다.

### 3.2 컨텍스트 압축으로 인한 세션 재시작

- **사실**: v6 작업 중 컨텍스트 한도에 도달하여 세션이 압축 재시작되었다. P5-3 E2E 수정 작업이 미완료 상태로 세션이 전환되었다.
- **영향**: 재시작 후 E2E 테스트를 재실행하여 33/33 통과를 재확인했다. 미커밋 파일이 남아있어 재시작 비용이 낮았다.
- **교훈(R18)**: 컨텍스트 재시작은 피할 수 없다. **미완료 작업의 중간 상태를 파일로 유지**(커밋이 아니더라도)하면 재시작 비용을 낮출 수 있다.
