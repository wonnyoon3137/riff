## 1. v0.3 → v0.4 무엇이 달라졌나

v0.3는 "새 도메인(Artist) 도입과 기존 토대 확장"이었다. v0.4는 **"v3 부채를 실측으로 상환하고, Artist 데이터 위에 첫 사용자 기능(아티스트 필터)을 출하한 사이클"**이다.

### 1.1 새로 도입한 것

- **품질 보강 트랙(Q)**: v3 교훈(R3/R4/R5) 3건을 실측/E2E로 상환. 가정이 아닌 데이터로 검증.
- **아티스트 필터(P3)**: KOPIS에 없는 출연진 검색을 자체 Artist DB 교차 필터로 구현. 자동완성 UI, BFF 교차 필터, URL 동기화, D8 상태 복원 전체 스택.
- **프로세스 정상화**: v3에서 위반했던 이슈→브랜치→PR 워크플로를 v4에서 전 작업에 적용.

## 2. 잘한 것

### 2.1 R1~R6 교훈 거의 완전 반영

| 교훈 | v4 반영 | 준수 |
|---|---|---|
| **R1** 이슈→브랜치→PR 자동 반사 | 9개 PR 모두 이슈 기반 + feature branch | ✅ (경미한 위반 1건 아래) |
| **R2** PR ~500줄 이하 | 최대 399줄(P3-3). 전 PR 500줄 이하 | **완전 준수** |
| **R3** E2E 보충 | Q-1에서 구군 4 + 가격 2 = 6건 작성 | **완전 이행** |
| **R4** MB 실 데이터 매칭률 | Q-2에서 23건 실측, 100% 매칭률. P3 게이트로 활용 | **완전 이행** |
| **R5** 구군 KOPIS 실 호출 | Q-3에서 10개 구군 전량 PASS | **완전 이행** |
| **R6** v4 범위 결정 | V-10 결정 포인트로 구조화 | **이행** |

v3에서 PR 1개에 4402줄을 넣었던 것이 v4에서는 9개 PR, 최대 399줄로 극적으로 개선되었다.

### 2.2 품질 게이트 → 기능의 자연스러운 흐름

Q-2(MB 실측)를 P3의 품질 게이트로 설정한 것이 효과적이었다. 매칭률 100%로 게이트를 통과한 뒤 P3을 자신 있게 착수할 수 있었다. "실측 전 가정 금지" 원칙이 작동한 사례.

### 2.3 BFF 교차 필터 설계

KOPIS에 출연진 검색 파라미터가 없다는 제약을 정직하게 인정하고, Artist DB → mt20id Set → BFF 교차 필터라는 깔끔한 우회로를 설계했다. 기존 필터(기간/지역/장르/공연장/검색)와 AND 결합이 자연스럽게 작동한다.

### 2.4 PR 크기 분해

P3-1(BFF, 42줄) + P3-2(API, 192줄) + P3-3(UI, 399줄) + P3-4(E2E, 342줄)로 분해해 각 PR이 리뷰 가능한 크기를 유지했다. 특히 P3-1이 42줄로 끝난 것은 기존 repo 패턴 재사용의 효과.

## 3. 못한 것 / 교훈

### 3.1 main 직접 커밋 1건 (R1 경미한 위반)

- **사실**: `d29d09a docs: update 0004` 커밋이 이슈/브랜치/PR 없이 main에 직접 push됨.
- **원인**: "docs 업데이트니까 괜찮겠지"라는 판단. docs-only라서 위험도는 낮지만 R1 원칙과 불일치.
- **교훈(R7)**: **docs-only여도 이슈→브랜치→PR 원칙은 동일.** 예외 없이 적용해야 습관이 된다. 단, exec plan 진행 로그 같은 사소한 업데이트는 다른 PR에 piggyback하는 것이 현실적.

### 3.2 P3-4 E2E 유실 — 이전 세션 작업물 미커밋

- **사실**: 이전 세션에서 `artist-filter.spec.ts`를 작성했으나 커밋하지 않은 채 컨텍스트 초과. 다음 세션에서 "이미 존재한다"고 오판하여 #60을 즉시 닫았다가, 검증 시 파일 부재 발견.
- **원인**: 세션 경계에서 working tree 상태를 검증하지 않음. `git status`를 먼저 확인했어야 했다.
- **교훈(R8)**: **세션 시작 시 `git status` 확인 필수.** 이전 세션의 미커밋 작업물이 있을 수 있다. "탐색 에이전트가 파일을 봤다"와 "파일이 커밋되었다"는 다른 것.

### 3.3 중복 이슈 (#55, #60)

- **사실**: P3-4에 대해 #55(이전 세션)와 #60(현재 세션) 두 개의 이슈가 생성됨. #55가 열린 채 방치.
- **교훈(R9)**: **이슈 생성 전 기존 이슈 검색.** `gh issue list --search "P3-4"` 등으로 중복 방지.

### 3.4 screens.md 미갱신 (G-4 부분 불이행)

- **사실**: G-4 게이트에서 `screens.md` S1에 아티스트 필터 UI 정의를 추가하도록 명시했으나, 실제로 `screens.md`는 변경되지 않음.
- **교훈(R10)**: **docs 게이트 체크리스트를 PR 단위로 검증.** 게이트를 선언만 하고 검증하지 않으면 의미 없다.

### 3.5 exec plan 진행 로그 지연 갱신

- **사실**: P3-0~P3-4 완료 기록, 이슈 번호 backfill이 v4 종료 시점에야 일괄 반영됨. plan 상태도 "draft"로 남아 있었음.
- **교훈(R11)**: **각 PR 머지 시 plan 진행 로그도 같은 PR에 포함.** 사후 일괄 갱신은 누락 위험.
