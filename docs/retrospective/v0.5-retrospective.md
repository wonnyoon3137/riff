## 1. v0.4 → v0.5 무엇이 달라졌나

v0.4는 "탐색 도메인 완결(아티스트 필터)"이었다. v0.5는 **"기존 사용자 없음 가정을 완전히 역전하고, 소셜 OAuth 인증 토대를 구축한 사이클"**이다.

### 1.1 새로 도입한 것

- **Phase 1 부채 정비**: v4 회고 교훈 R10/R12/R13 전량 상환. screens.md UI 정의 보충, lint 0 errors 0 warnings, tech-debt-tracker 현행화.
- **P4 인증 스택**: Auth.js(NextAuth v5) + SQLite adapter + JWT 세션. 카카오/구글 OAuth Provider. User/Account/Session DB 스키마.
- **로그인 UI(S3)**: `/login` 페이지, AppBar 헤더 프로필/로그아웃 드롭다운.
- **Edge 호환 분리 패턴**: `auth.config.ts`(Edge Runtime용) vs `auth.ts`(Node.js 전용) 분리. Next.js middleware에서 `node:crypto`/`better-sqlite3` 오류 해결.

### 2.2 Auth.js Edge/Node.js 분리 패턴 발견

middleware.ts에서 `auth.ts`를 직접 import하면 `node:crypto`와 `better-sqlite3`가 Edge Runtime 번들에 포함되어 오류가 발생했다. 이를 `auth.config.ts`(providers만, Node.js 없음)와 `auth.ts`(adapter 포함, Node.js 전용)로 분리하여 해결했다. 이 패턴은 Auth.js v5 공식 권장 패턴이기도 하며 향후 P5/P6에서도 재사용된다.

### 2.3 Venue/Artist repo 패턴 세 번째 재사용

User DB(`src/server/users/repo.ts`)를 Venue/Artist repo 패턴 그대로 구현: WAL 모드, 모듈 싱글톤, `:memory:` 테스트 주입. 새 패턴을 설계할 필요 없이 기존 코드를 인터페이스에 맞게 조립했다. 단위 테스트 16건도 `venues-repo.test.ts` 구조를 재사용하여 빠르게 작성.

### 2.4 Rate limit 재시작 후 컨텍스트 복원

이전 세션이 `API Error: Rate limit reached`로 중단되었으나, 재시작 후 exec-plan 진행 로그와 Git 상태(`git status` + `git log` + `gh issue list`)만으로 정확히 재개 지점(P4-1 착수 전)을 파악했다. 세션 단위로 PR을 완결짓는 워크플로가 재시작 비용을 최소화했다.

### 2.5 PR 크기 500줄 이하 유지

P4-2(user DB + adapter)가 가장 컸으나 ~370줄 이내. P4-1, P4-3은 각각 120, 15줄로 작게 유지. 인증이라는 복잡한 스택을 5개 PR로 명확히 분해했다.

## 3. 못한 것 / 교훈

### 3.1 Edge/Node.js 오류를 P4-3이 아닌 P4-5에서 발견

- **사실**: middleware.ts가 auth.ts를 import하는 순간(P4-3)에 Edge 오류가 잠재했으나, E2E 서버를 실제로 띄운 P4-5(E2E 테스트)에서야 `node:crypto UnhandledSchemeError`가 드러났다.
- **원인**: `pnpm typecheck`와 `pnpm lint`는 번들링 오류를 잡지 못한다. 실제 Next.js 서버를 띄워야 webpack 번들링 오류가 노출된다.
- **교훈(R14)**: **Auth.js + DB adapter 설정 시 P4-1/P4-2 단계에서 `auth.config.ts` 분리 패턴을 즉시 적용.** middleware 작성 단계(P4-3)에서 dev 서버를 띄워 번들링 오류를 조기 확인.

### 3.2 E2E 픽스처 데이터 오해

- **사실**: `비로그인 회귀 방지` 테스트에서 `서울 공연 A`(REGION_ITEMS에만 있음)를 전국 기본 목록에서 찾으려 해 테스트 2건이 실패했다. 픽스처 파일을 읽지 않고 다른 E2E 테스트의 코드만 참조하여 복붙한 결과.
- **교훈(R15)**: **E2E 테스트 작성 전 `e2e/fixtures/performances.ts`의 데이터 구조를 확인.** 전국 기본 목록(`NATIONWIDE_PAGE_1`)과 지역 목록(`REGION_ITEMS`)의 항목명이 다르다.

### 3.3 세션 분절 (Rate limit)

- **사실**: P4-1 도중 Rate limit으로 세션이 강제 종료되었다. backend-dev 서브에이전트가 17 tool uses 진행 후 중단. 작업 결과가 미커밋 상태로 유실되었다.
- **교훈(R16)**: Rate limit은 외부 요인이지만, **작업 단위를 더 작게 분해하거나 서브에이전트 대신 직접 작업하면 재시작 비용이 낮아진다.** 서브에이전트가 17 tool uses 동안 커밋하지 않은 것이 유실의 직접 원인. PR 단위 커밋을 서브에이전트 완료 조건으로 명시해야 한다.
