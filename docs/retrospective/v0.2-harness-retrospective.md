## 1. v0.1 → v0.2 무엇이 달라졌나

v0.1은 "하네스로 MVP를 만들 수 있는가"의 증명이었다. v0.2는 **"하네스로 반복 가능한 개발 사이클을 돌릴 수 있는가"** 의 증명이다. v0.1 회고가 남긴 권장사항(P1~P5)을 실제로 실천하고, 프로세스를 코드/문서/CI로 고정했다.

### 1.1 v0.1 권장사항의 이행

| v0.1 권장 | v0.2 이행 |
|---|---|
| **P1 하네스 일관 적용** | 전 작업을 orchestrator → planner → dev → qa 파이프라인으로. "작아서 생략" 없음(단 2줄 수정 등 초소형은 명시적 직접 처리) |
| **P2 커밋 세분화** | 논리 단위 커밋 + `type(scope): 요약 (#이슈)` + D# 태그. `CONTRIBUTING.md`로 규칙 고정(#6) |
| **P3 E2E 테스트** | **M1 최우선**으로 Playwright 3종 착수 → 곧바로 결함 5건 발견(아래 4.1) |
| **P4 QA 보고서 승격** | qa 검증을 인라인 보고로 운용(별도 `_workspace` 미사용). 대신 QA 티어를 `docs/process/`로 명문화(#5) |
| **P5 중간 체크포인트** | 사람 결정 게이트를 명시적으로 운용: R5(구군 범위)·DEC-A(저장소)·DEC-B(동기화 주기)·DEC-S1(검색 기간) |

### 1.2 새로 도입한 것

```
기능 브랜치 → PR → CI(verify · e2e · docs-guard) → squash 머지   (#6)
├── docs↔code 드리프트 가드 CI                                    (#4/#20)
├── QA 게이트 티어링 Fast/Deep                                    (#5)
├── 취향/구조 린트 강제(a11y·hex·레이어 의존·미검증코드)          (#2/#18)
└── ESLint CLI flat config (next lint deprecated 탈출)            (#11)
```

## 4. 무엇이 잘 되었나

### 4.1 E2E 회귀 안전망이 즉시 값을 했다

M1에서 E2E 3종을 깔자마자 **결함 5건(#21~#25)** 이 드러났다. 특히:
- **#21 다중지역 병합 누락** — page>1에서 항목이 통째로 빠지는 **production 버그**. 무한스크롤 다음 페이지가 비었다. 측정 하네스가 90건 누락을 정량 검출.
- #22 debounce 네비 레이스, #23 스크롤 복원 — D8 상태보존의 실제 결함.

전부 추적 → 수정 → "E2E가 결함을 회피하며 통과하던 것"을 "실제로 잡는 테스트"로 전환했다. **회귀 안전망이 없었다면 M2~M3 리팩터링 중에 묻혔을 버그들이다.**

### 4.2 docs-우선이 범위 이탈을 막았다

검색(#8)은 D10="검색 v0.1 제외"와 충돌하는 작업이었다. planner가 **코드 착수 전 D10/F5를 먼저 갱신**(R1 게이트)하도록 강제했고, 그 과정에서 KOPIS 파라미터 조합 제약을 확인해 "별도 호출 경로 불요, AND 결합"이라는 올바른 설계가 나왔다. 문서를 건너뛰었다면 잘못된 결합 가정으로 재작업했을 것이다.

### 4.3 실측이 추측을 대체했다

KOPIS rate limit은 v0.1 내내 "미검증→실측" 상태였다. v0.2에서 키 주입 후 실측해 **안전 동시성 상한 5**를 확정했고, 이 값이 병합 batch(#21)·동기화(#9)·캐시(#17) 결정의 일관된 근거가 됐다. "확인 필요"가 코드에 박힌 매직넘버로 굳지 않았다.

### 4.4 QA 티어링이 속도와 안전을 양립시켰다

큰 기능(M2 DB·M3 검색)은 qa-engineer Deep 검증(경계면+결정 감사)으로, 작은 버그픽스는 Fast(E2E+CI)로 처리했다. qa가 M3에서 잡은 F5.3 결함(검색 단독 시 초기화 버튼 비활성)은 **같은 PR에서 즉시 수정**해 백로그로 새지 않았다.

### 4.5 프로세스를 코드/CI로 고정했다

PR 워크플로·드리프트 가드·린트 강화로 "다음에도 지켜질 것"을 보장했다. 특히 린트 강화(#2/#18)는 a11y·hex·레이어 의존·미검증 코드값을 **CI가 자동 차단**하게 만들어, 취향이 사람 리뷰가 아니라 도구로 강제된다.

## 5. 무엇이 아쉬웠나

### 5.1 로컬 lint가 늦게 복구됐다

`next lint` + `eslint-config-next`가 Node 25에서 비호환(`@rushstack/eslint-patch`)이라 **로컬 lint가 M2 중반까지 안 돌았다**. CI(Node 20)에만 의존하던 기간이 있었고, #11로 flat config 마이그레이션을 한 뒤에야 로컬에서 `eslint .`가 돌았다. → **개선**: 환경 호환성(런타임 버전)을 v0.3 초반에 점검.

### 5.2 npm/pnpm 혼선

`package-lock.json`(npm 흔적)이 세션 내내 untracked로 남아 frozen-lockfile/lint 흐름에 잡음을 줬다. 뒤늦게 삭제·gitignore. → **개선**: 부트스트랩 시 패키지 매니저 단일화 가드(이미 `packageManager` 필드 존재, lockfile 가드 추가).

### 5.3 `.claude/` 자기수정 마찰

스킬·에이전트 파일(`kopis-integration` 실측 반영, QA 티어 반영) 수정이 자동 차단돼 매번 사람 승인이 필요했다. 안전장치로서는 옳지만, "실측 결과를 스킬에 반영" 같은 정당한 갱신도 흐름을 끊었다. → **개선**: docs를 SSOT로 두고 `.claude`는 링크만 두는 패턴(이번에 부분 적용)을 표준화.

### 5.4 ROI 정량 측정 미완 (#7)

with/without orchestration A/B를 하려면 측정용 신규 기능이 필요한데, v0.2 작업이 모두 끝나 남지 않았다. 관찰신호(결함 5건 발견·머지 전 포착·범위이탈 차단 1건)와 측정틀만 문서화하고 **정량 실측은 v0.3으로 이관**.

## 7. 하네스 구조 자체에 대한 피드백 포인트

v0.3 착수 전 사용자(혜원)가 검토할 항목:

1. **QA 티어 판정 정확도**: Fast/Deep 5축 기준(`qa-gate-tiers.md`)이 실제로 위험을 잘 갈랐는가? Fast로 보낸 작업에서 놓친 게 없었는가?
2. **planner 위임 비용**: 작은 작업까지 planner를 거치는 게 과한가? "직접 처리" 임계(이번엔 ~2줄)를 어디에 둘 것인가?
3. **드리프트 가드 강도**: 거짓양성 0을 위해 의도적으로 약하게 잡았다(자연어·교차링크 제외). 더 강하게 갈 가치가 있는가?
4. **실측의 docs 반영 경로**: `.claude` 자기수정 차단을 고려할 때, 실측 결과를 어디에 두는 게 맞는가(docs SSOT vs 스킬)?
5. **회고 주기**: v0.1→v0.2는 회고가 다음 버전 이슈의 입력이 됐다. 이 루프(회고 → 이슈 → 작업 → 회고)를 v0.3에서도 유지할 것인가?
