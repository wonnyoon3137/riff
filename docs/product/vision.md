## 1. 비전 — 4도메인 정의

Riff의 최종 형태는 "공연 정보 탐색기"를 넘어 **공연을 좋아하는 사람의 라이프사이클 전체**를 담는다: 발견 → 팔로우 → 디깅 → 기록.

### ① 탐색 (Explore) — "한 곳에서 필터링하며 공연을 찾는다"
- 필터: 아티스트별 / 공연장별(지역별) / 장르별 / 가격대별 / 월별
- 공연 상세: 공연장 정보, 티켓 가격, 예매 링크
- 데이터: KOPIS API + **자체 수집 보강**
- → 현재 v0.1/v0.2의 핵심. 확장 대상은 **아티스트별·가격대별 필터**.

### ② 관심 아티스트 (Following) — "좋아하는 아티스트를 팔로우하고, 그 기반으로 새 아티스트를 발견한다"
- 관심 아티스트 지정/관리
- 관심 아티스트 기반 추천 아티스트
- 관심 아티스트 공연 알림 (향후)
- → **신규 도메인.** Artist 엔티티 + 사용자 계정이 전제.

### ③ 디깅 (Digging) — "아티스트를 더 깊이 파는 공간"
- 관심 아티스트 기반 음악 디깅
- 사람들이 올린 플레이리스트 모음 (YouTube 연동)
- 관련 아티스트 앨범 추천 공유
- 음악 덕후 커뮤니티
- → **신규 도메인.** Artist 엔티티 + 인증 + 외부 연동(YouTube) + UGC/커뮤니티가 전제. 가장 무겁다.

### ④ 마이페이지 (My Record) — "본 공연이 기록으로 쌓인다"
- 공연 관람 기록 (체크인 또는 자동 기록)
- 영상·이미지 업로드 → 인스타 피드 형태 아카이빙
- 공연 히스토리 타임라인
- → **신규 도메인.** 인증 + UGC 스토리지가 전제.

> **비전과 기존 overview의 관계:** overview §6 로드맵이 이미 v0.2(자체 인덱스/아티스트·가격 파싱), v0.3(인증·관심 아티스트·공연 기록), v0.4+(디깅·UGC·추천)를 스케치했다. 본 비전은 그 스케치를 4도메인으로 **구체화·정합**한 것이다. 단, overview는 "아티스트별 필터=v0.2"로 적었으나, 아래 §3.1의 데이터 난점 때문에 **현실적으로 v0.2(구현 완료)에 들어가지 못했고** 본 비전에서 Artist 엔티티 트랙으로 재배치한다.

### 2.2 한 줄 요약

**① 탐색의 "공연 중심" 부분은 거의 완성**(목록·5필터 중 3.5개·상세·검색·상태복원·자체 공연장 DB·E2E 안전망). 미구현은 **① 탐색의 두 필터(아티스트·가격대)** 와 **②③④ 도메인 전부**다. ②③④는 단순 기능 추가가 아니라 **새 엔티티 + 인증 + 외부 연동/스토리지**라는 토대 전환을 요구한다.

### 2.3 현재 엔티티 vs 비전이 요구하는 엔티티

`data-model.md` §2 현재 엔티티: `Performance`(비저장/조회), `Venue`(자체 DB 저장), `BookingRelate`, `IntroImage`, `FilterState`/`ListState`(클라이언트). **사용자·아티스트·기록 개념이 전혀 없다.**

비전이 새로 요구하는 엔티티(추정 — 확정 아님):

| 신규 엔티티 | 도메인 | 핵심 난점 |
|---|---|---|
| `Artist` | ①②③ | KOPIS에 식별자 없음 → 자체 식별·정규화 필요(§3.1) |
| `Artist↔Performance` 출연 관계 | ①② | `prfcast` 자유 텍스트에서 추출해야(§3.1) |
| `User` / 인증 | ②③④ | "로그인 없음" 결정 뒤집기(§3.2) |
| `Follow` (User↔Artist) | ② | User·Artist 둘 다 전제 |
| `Record` (관람 기록) | ④ | User + 공연 식별. 자동 vs 수동(§6) |
| `Upload` (영상/이미지) | ④ | 스토리지·모더레이션·비용(§3.4) |
| `Playlist` (YouTube 링크 모음) | ③ | 외부 API·UGC(§3.3) |
| `Post`/`Comment` (커뮤니티) | ③ | 백엔드 대확장·운영(§3.5) |

> 또한 현재 **v0.1 비저장 원칙**(data-model §1.5: 공연은 저장 안 하고 매 요청 KOPIS 조회, Venue만 예외)이 ②④와 충돌한다. Follow/Record가 특정 공연·아티스트를 **영속 참조**하려면, 그 시점의 공연/아티스트 식별자를 안정적으로 저장해야 한다. 비저장 원칙의 부분 완화가 필요하다.

## 3. 근본 제약 · 리스크 (가장 중요)

> 비전의 난이도는 기능 수가 아니라 **토대 전환** 4가지에서 나온다. 아래는 docs 근거 기반의 정직한 평가다.

### 3.1 [Top 1] 아티스트 엔티티 문제 — ②③의 토대이자 최대 난점

**사실(근거):**
- KOPIS는 **공연 중심** 데이터다. 출연진은 `prfcast` **단일 자유 텍스트 필드**로만 제공된다(`kopis-integration.md` §정규화 "자유 텍스트 원문 보존(파싱 금지): prfcast…", `data-model.md` §3.2 `cast?: string`). 검증 결과 KOPIS 어디에도 **아티스트 식별자·구조화 출연진 목록이 없다.**
- 즉 KOPIS만으로는 "아티스트"라는 1급 개념을 만들 수 없다. "아티스트별 필터/팔로우/추천"(①의 아티스트 필터, ②, ③ 전부)은 **아티스트를 식별·정규화하는 자체 데이터**가 있어야 성립한다. 사용자가 비전에서 직접 "자체 수집 보강"을 언급한 지점이 바로 여기다.

**난점의 본질:**
- `prfcast` 텍스트에서 사람/팀 이름을 추출(NER)하고, "방탄소년단" vs "BTS" vs "防弾少年団" 같은 **표기 변형을 동일 아티스트로 정규화(엔티티 해소)** 해야 한다. 출연진 텍스트에는 배역·역할·구분자가 뒤섞여 있어 단순 파싱이 아니다.
- 한번 만든 아티스트 ID를 공연마다 **안정적으로 연결**해야 추천/팔로우가 작동한다.

**데이터 소스 후보와 난이도(정직한 평가 — 어느 것도 공짜가 아님):**

| 소스 | 방식 | 난이도 | 한계 |
|---|---|---|---|
| (a) `prfcast` 자체 추출 | NER + 정규화 파이프라인 | **높음** | 노이즈 큼, 표기 변형, 검증/교정 운영 필요. 정확도 보장 어려움 |
| (b) 외부 음악 메타 API (예: Spotify/MusicBrainz 등) | 아티스트 마스터를 외부에서 가져와 매칭 | **중~높음** | API 약관·쿼터, 공연 출연진과의 매칭 정확도, 클래식/연극 등 비대중음악 커버리지 |
| (c) 수동 큐레이션 | 사람이 아티스트 마스터 구축·연결 | **중(품질 높음)** | 확장성 낮음, 운영 인건비. 초기 시드로는 현실적 |
| (d) 하이브리드 | 인기 아티스트는 수동 시드 + 나머지는 (a)/(b) 자동, 사람 검수 | **중~높음** | 가장 현실적이나 운영 체계 필요 |

→ **이것이 비전 전체의 임계 경로다.** Artist 엔티티 없이는 ②③가 시작도 못 한다. 그리고 Artist는 단발 작업이 아니라 **지속 운영되는 데이터 파이프라인**이다.

### 3.2 [Top 2] 인증/계정 — ②③④ 전부의 전제, "로그인 없음" 결정의 역전

**사실(근거):**
- 현재 명시적 결정: overview §5 "사용자 인증/로그인 = v0.3+, 로그인이 필요한 기능 자체가 없음". screens §82 "상세 페이지에 '관심/저장' 같은 **로그인 의존 액션 없음**". data-model에 `User` 엔티티 부재.
- ②(팔로우)·③(커뮤니티/공유)·④(개인 기록/업로드)는 **전부 사용자 계정 전제**다. "이 사용자가 이 아티스트를 팔로우", "이 사용자의 관람 기록"이 성립하려면 사용자 식별이 필수.

**의미:**
- 인증 도입은 단순 기능이 아니라 **제품 성격 전환 + 기술 토대 추가**다: 회원가입/로그인(소셜 OAuth 등), 세션/토큰, 개인정보 처리방침·약관, 계정 데이터 스토어, 권한 모델.
- 현재 아키텍처(BFF + 비저장 + 공연장만 SQLite 파일)에 **사용자 데이터 영속 계층**이 새로 들어온다. 이것이 ②④의 공통 선행 인프라.

### 3.3 [Top 3] 외부 연동 — 디깅의 YouTube, 그리고 예매 링크

- ③ 디깅의 "플레이리스트 모음(YouTube 연동)", "앨범 추천"은 **외부 API**(YouTube Data API 등)에 의존한다. **API 키·쿼터·약관**(임베드/표시 정책) 검토가 선행돼야 한다. KOPIS rate limit을 실측해 상한 5로 확정한 선례(회고 §4.3)처럼, **추측 금지·실측** 원칙이 동일하게 적용돼야 한다.
- 예매 링크(`relates`)는 이미 ①에서 외부 이동으로 사용 중. 디깅의 외부 연동은 그보다 한 단계 깊은 통합(검색·임베드·메타 조회).

### 3.4 UGC / 스토리지 — ④ 영상·이미지 업로드

- ④의 "영상·이미지 업로드 → 인스타 피드 아카이빙"은 **바이너리 스토리지(S3 등) + CDN + 비용**, 그리고 **모더레이션**(불법/저작권 콘텐츠 차단)을 요구한다. 현재 인프라(파일 SQLite)와는 차원이 다른 운영 부담.
- 영상은 특히 용량·트랜스코딩·재생까지 고려해야 해, ④ 내에서도 **이미지 먼저 / 영상 나중** 분리가 현실적.

### 3.5 커뮤니티 — ③ 공유/덕후 커뮤니티

- ③의 "커뮤니티"는 게시/댓글/신고/모더레이션 = **백엔드 대확장 + 지속 운영 부담**(스팸·악성 대응). 비전에서 가장 무거운 단일 항목. MVP에서 "커뮤니티"를 어디까지로 볼지(읽기 전용 큐레이션 vs 풀 UGC 커뮤니티)는 사람 결정 사안(§6).

## 4. 단계적 로드맵 (의존 순서)

> 원칙: **무엇이 무엇의 전제인지**로 phase를 가른다. 각 phase는 선행 인프라/결정을 명시한다. phase 번호는 제품 버전과 1:1이 아니다(여러 phase가 한 버전에 들어갈 수도, 한 phase가 여러 버전에 걸칠 수도 있다).
>
> **✅ v3 확정(2026-06-12, V-1):** **v3 = P1(① 탐색 강화) + P2(Artist 엔티티, 하이브리드) 병행.** 두 트랙은 서로 독립(의존 0)이며 인증을 요구하지 않는다. 실행계획 `docs/exec-plans/active/0003-v3-plan.md`. 인증(P4)·Following(P5)·④/③은 v3 범위 밖(phase 직전 결정).

### 4.3 핵심 의존 관계 (명시)

- **Artist 엔티티(P2) → Following(P5) → 추천(P7), 그리고 → 아티스트 필터(P3)·디깅(P9).** Artist 없이는 ②③의 거의 전부가 막힌다.
- **인증(P4) → Following(P5) · My Record(P6) · 디깅 커뮤니티(P9).** 사용자 개념 없이는 ②④와 ③의 커뮤니티가 막힌다.
- **My Record 기록(P6) → 업로드(P8).** 기록이 먼저, 무거운 UGC 스토리지는 그 위에.
- **디깅(P9)은 거의 모든 토대(Artist·인증·외부API·UGC·커뮤니티) 위에 선다 → 마지막.**
- **① 탐색 강화(P1)는 어느 것에도 의존하지 않는다** → 즉시 착수 가능(아래 §5).

## 5. v3 첫 범위 (✅ 확정 — V-1)

> 기준: **(가치 × 의존도 역수)**. 의존이 적고 기존 토대를 재사용하며 사용자(혜원)의 핵심 페인을 직접 푸는 것.
> **✅ 2026-06-12 사람 결정(V-1): 아래 §5.1 추천안(P1+P2 병행)을 v3 첫 범위로 확정.** 실행계획은 `docs/exec-plans/active/0003-v3-plan.md`로 이관 완료.

### 5.1 확정: **두 트랙 병행 — "P1 ① 탐색 강화"를 먼저 출하하고, "P2 Artist 엔티티"를 토대 투자로 동시 착수**

**근거:**

1. **P1(① 탐색 강화)은 의존 0, 가치 직접.**
   - 가격대 필터·구군 드릴다운은 인증도 Artist도 필요 없다. 기존 KOPIS+BFF+필터 토대를 그대로 확장한다(D2~D9·F2 재사용).
   - 가격대 필터는 overview §5에서 이미 "v0.2"로 약속됐던 미상환 항목이고, `pcseguidance` 파싱이라는 명확한 작업이다. 빠른 사용자 가치.
   - **단, 가격 파싱은 Risk 2의 자유 텍스트 문제** — v0.1에서 의도적으로 미룬 것이다. "VIP 99,000 / R 88,000" 같은 다양한 표기에서 가격대를 추출하는 규칙은 만들 수 있으나 100% 정확하지 않음을 전제로, "대략적 가격대 밴드(예: 무료/~3만/~7만/7만+)" 수준의 보수적 설계를 권한다.

2. **P2(Artist 엔티티)는 비전의 임계 경로이므로, 빨리 시작할수록 ②③ 전체가 앞당겨진다.**
   - Artist는 단발 기능이 아니라 **데이터 파이프라인 + 품질 운영**이라 리드타임이 길다. v3에서 "①에 보이는 작은 가치"(예: 상세 페이지에서 출연 아티스트를 클릭 가능한 칩으로, 아직 팔로우는 없이)부터 시작해 데이터를 쌓는 게 현실적이다.
   - 단, **P2는 §6의 "아티스트 데이터 소스" 결정 없이는 착수 불가.** 이 결정이 v3의 첫 게이트다.

**왜 인증(P4)/Following(P5)을 v3 첫 범위로 추천하지 않는가:**
- 인증은 큰 제품 전환(약관·개인정보·세션)이고, Following은 인증 + Artist **둘 다** 전제다. 의존이 가장 깊어 첫 출하로는 리스크가 크다. Artist 데이터가 어느 정도 쌓이고 인증 결정이 난 **다음** 버전이 적합.

### 5.2 대안 (사람이 다른 가치를 우선한다면)

- **대안 A — "내 기록부터":** ④ My Record(체크인+타임라인)는 Artist 없이 인증(P4)만으로 가능. "본 공연 기록"은 감정적 가치가 크고 재방문을 만든다. Artist 파이프라인의 긴 리드타임을 피하고 싶다면 **인증+My Record**가 v3 후보. (단 인증이라는 큰 결정이 선행.)
- **대안 B — "탐색만 더 깊게":** ②③④를 모두 미루고 P1만 충실히(가격·구군·정렬·검색 고도화). 토대 전환 없이 ① 탐색기로서 완성도를 끝까지. 가장 안전하나 비전 확장은 정체.

> 어떤 것을 v3 첫 범위로 할지는 **사람 제품 결정**이다(§6 우선순위). 본 문서는 "P1은 즉시 가능 + P2는 임계 경로라 빨리 씨를 뿌릴 가치"라는 근거만 제공한다.
