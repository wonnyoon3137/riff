## F1. 공연 목록 탐색

### F1.1 공연 카드 목록 표시
- 사용자가 메인 페이지에 진입하면 공연 목록을 카드 그리드 형태로 표시한다.
- **첫 진입 시 기본 노출**: 오늘부터 30일 이내 + 전국 + 전체 장르
- **카드 표시 항목** (KOPIS `pblprfr` 응답 기준)
  - 포스터 이미지 (`poster`)
  - 공연명 (`prfnm`)
  - 공연 기간 (`prfpdfrom` ~ `prfpdto`)
  - 공연장명 (`fcltynm`)
  - 지역 (`area`)
  - 장르 (`genrenm`)
  - 공연 상태 뱃지 (`prfstate`: 공연중/공연예정/공연완료)

### F1.2 페이지네이션 — 무한 스크롤
- 스크롤이 하단에 가까워지면 (Intersection Observer) 다음 페이지 자동 로드
- 페이지당 30개 (KOPIS는 최대 100개까지 가능)
- **상태 복원 요구사항 (F1.2 ↔ F3.3 연계)**
  - 상세 페이지 진입 후 뒤로가기 시 스크롤 위치 및 로드된 페이지 수 복원
  - 구현 방안은 화면설계서 / 데이터 모델 단계에서 확정 (예: sessionStorage, React Query 캐시 활용)
- **URL 동기화**
  - 필터 상태는 쿼리스트링에 반영 (공유 가능)
  - 페이지 번호는 URL에 미포함 (무한 스크롤 특성상). 단, 직접 진입 시 deep link로 특정 페이지 복원은 미지원.

### F1.3 로딩 / 빈 상태 / 에러 처리
- **로딩**
  - 첫 로드: 스켈레톤 카드 그리드
  - 추가 로드 (무한 스크롤): 하단에 스피너
- **빈 상태**: "조건에 맞는 공연이 없습니다" 메시지 + 필터 초기화 버튼
- **에러**: "공연 정보를 불러오지 못했습니다" 메시지 + 재시도 버튼

## F2. 필터링

### F2.1 월별 필터
- 이번 달 / 다음 달 / 그 다음 달 / 직접 선택 (날짜 범위 피커)
- KOPIS `stdate`, `eddate` 파라미터에 매핑
- **31일 초과 시**: UI에서 31일 이내로 제한 (v0.1)
  - 사용자가 31일을 초과해 선택하면 토스트로 "최대 31일까지 선택 가능합니다" 안내
  - v0.2에서 분할 호출 방식으로 확장 예정

### F2.2 지역 필터 — 다중 선택
- 시도 단위 1단계 → 구군 단위 2단계 (드릴다운)
  - **v0.2까지는 시도 단위만 구현**(실행계획 0002 R5). **v3에서 구군 드릴다운 재개.**
- KOPIS `signgucode` (시도), `signgucodesub` (구군) 매핑
- **다중 선택 허용** — 시도 또는 구군을 여러 개 선택 가능
- KOPIS API는 단일 지역 코드만 받으므로, 다중 선택 시 클라이언트에서 병합 처리
  - 구현 방안: 선택된 지역별로 병렬 API 호출 후 결과 병합 (BFF 레이어에서 처리)
  - 데이터 모델 / API 명세 단계에서 상세화
- "전국" 선택 시 다른 지역 선택 해제

### F2.3 장르 필터
- KOPIS `shcate` 코드 매핑 (예: AAAA=연극, BBBB=뮤지컬, CCCA=대중음악 등)
- 칩(chip) 형태로 다중 선택 허용
- 장르 코드 한글 매핑은 별도 문서(`docs/api/kopis-codes.md`)에서 관리

### F2.4 공연장 필터 — 사전 동기화
- 공연장명 자동완성 입력
- 선택 시 KOPIS `prfplccd` 파라미터로 전달
- **공연장 마스터 데이터**: KOPIS `prfplc` (공연시설 목록 조회 서비스)에서 사전 동기화하여 자체 DB에 저장
  - 동기화 방식: 초기 일괄 fetch + 주기적 갱신 (스케줄 빈도는 데이터 모델 단계에서 결정)
  - 이 자체 DB가 v0.2 자체 인덱스 구축의 기반이 됨

### F2.5 필터 조합 동작
- **필터 변경 즉시 반영** (debounce 300ms)
- 모든 필터는 URL 쿼리스트링에 반영 (공유/북마크 가능)
- "필터 초기화" 버튼 제공

## F5. 공연명 검색 (v0.2 도입) · #8

> **결정 변경(D10):** v0.1에서는 검색을 제외했으나(자체 인덱스 부재 우려), KOPIS `/pblprfr` 목록 API에 **공연명 검색 파라미터 `shprfnm`이 존재함이 검증**되었다(`kopis-integration.md §2.1`). 자체 인덱스 없이도 KOPIS 위임 검색으로 v0.2에 도입 가능하므로 F5를 활성화한다. 변경 이력은 본 절 말미 결정 로그 참조.

### F5.1 검색 대상 / 범위
- **검색 대상: 공연명(`shprfnm`)만.** 출연진·줄거리·공연장명 등은 v0.2 검색 범위 아님.
  - (참고) `shprfnmfct`(공연시설명 검색)는 별도 파라미터로 존재하나 v0.2 미사용 — 공연장 지정은 기존 공연장 필터(F2.4, `prfplccd`)가 담당.
- 매칭 방식은 KOPIS 서버 측 부분일치에 위임(자체 인덱스 구축 전제 아님). 자체 인덱스 기반 고도화는 v0.3 이후.

### F5.2 기존 필터와의 결합 규칙 (AND)
- **모든 필터와 AND 결합.** 검색어 + 기간(F2.1/D1) + 지역(F2.2/D4) + 장르(F2.3) + 공연장(F2.4) + 정렬(F4)이 동시에 적용된다. 검색 시 **어떤 필터도 무시하지 않는다.**
- **KOPIS 파라미터 제약 확인 결과(중요):** `shprfnm`은 `/pblprfr` 목록 API의 **선택 파라미터**로, `shcate`/`signgucode`/`prfplccd`/`prfstate` 등 다른 선택 파라미터와 **동시 사용 제약이 없다**(`kopis-integration.md §2.1`, §4 시퀀스 다이어그램에서 `signgucode`+`shcate` 동시 전송 검증). 즉 검색은 **기존 호출에 `shprfnm` 한 파라미터를 추가**하는 것일 뿐, 별도 호출 경로가 필요 없다.
- **다중 지역(D4)과의 결합:** 다중 지역은 이미 지역별 병렬 호출 후 BFF 병합(D4) 구조다. `shprfnm`은 **각 지역 호출에 동일하게 부착**되어 결합된다(추가 호출 수 증가 없음, 병합 로직 불변).
- **기간(stdate/eddate) 필수 제약:** KOPIS는 `stdate`/`eddate`가 필수이므로 검색도 항상 기간 범위 안에서 수행된다. 검색이 기간 필터를 무시하지는 않는다. → **DEC-S1 확정(2026-06-12, 사람 결정): 옵션 A 순수 AND.** 검색은 현재 기간 범위 내에서만 수행하고, 기간 확장은 사용자가 기간 필터로 직접 제어한다. 검색이 기간 창을 자동 확장하지 않는다(D1 기본 30일·D3 31일 제약과 충돌 없음, 결합 규칙 일관). 옵션 B/C(자동 확장)는 31일 제약상 다중 호출 분할이 필요해 v0.2에서 미구현.

### F5.3 입력 동작
- **입력 UI:** S1 상단(상단바 또는 필터바 상단)의 검색 입력(Search/Text field). 상세는 `screens.md` §2.3a.
- **반영 방식: debounce 후 자동 재조회(300ms, D6 일관).** 별도 "검색" 제출 버튼 없이 입력 즉시 반영(필터 변경과 동일 UX). Enter 입력 시 debounce 대기 없이 즉시 재조회(선택).
- **URL 동기화:** 검색어를 쿼리스트링 `?q=`에 반영(공유/북마크 가능, F1.2/F2.5와 일관). 뒤로가기 복원 대상(D8)에 포함.
- **필터 초기화(F2.5):** "필터 초기화" 시 검색어도 함께 비운다.

### F5.4 가드 / 빈 상태
- **최소 길이 2자:** 공백 trim 후 2자 미만이면 `shprfnm` 미전송(전체 목록 = 검색 비활성 상태로 동작). 공연장 자동완성 `q<2` 가드와 일관.
- **빈 검색:** 검색어가 비면 `shprfnm` 파라미터를 빼고 기존 필터만으로 조회.
- **결과 0건:** 기존 S1-E(빈 상태) 재사용 — "조건에 맞는 공연이 없습니다" + 필터 초기화. (검색 전용 문구는 v0.2 단순화 차원에서 별도 추가하지 않음.)
- **공백 정규화:** 앞뒤 공백 제거. 내부 공백은 KOPIS에 그대로 전달(URL 인코딩).

### F5.5 캐싱
- 캐시 키(`perf:{filterHash}:{page}`)에 검색어를 포함한다(`filterHash`에 `shprfnm` 추가). 검색어가 다르면 다른 캐시 엔트리.

## F6. 가격 밴드 표시 (v3 P1-1) · QA 티어: Deep

> 상세 페이지(S2)에서 `pcseguidance` 자유 텍스트를 파싱하여 가격 밴드 칩으로 표시한다.

### F6.1 가격 밴드 정의

| 밴드 | 조건 |
|---|---|
| 무료/초대 | "무료", "초대" 등 키워드 감지 또는 금액 0원 |
| ~3만 | 최저가 30,000원 이하 |
| 3~7만 | 최저가 30,001~70,000원 |
| 7만+ | 최저가 70,001원 이상 |
| 가변/미정 | 파싱 실패 또는 가격 정보 부재 |

### F6.2 파싱 규칙
- `pcseguidance` 원문에서 금액(숫자+원/만원 등)을 추출하고, **최저가 기준**으로 밴드를 결정한다(보수적 — 다수 등급이면 가장 저렴한 가격 기준).
- **파싱 실패 시 "가변/미정"으로 분류한다**(거짓 제외 방지 — 파싱 오류로 공연이 누락되지 않도록).
- 원문 텍스트는 기존대로 함께 표시한다(밴드 칩은 보조 정보).

### F6.3 범위
- **표시만**: 상세 페이지(S2) 가격 영역에 밴드 칩을 추가 표시.
- **목록 필터는 v3 범위 밖**(V-4/DEC-P1 확정) — 밴드 기반 목록 필터링은 후속 phase에서 구현.

## F7. 출연 아티스트 표시 (v3 P2) · QA 티어: Deep

> 상세 페이지(S2)에서 출연 아티스트를 칩으로 표시한다. 자체 Artist 엔티티(data-model.md §3.5) 기반.

### F7.1 표시
- 상세 페이지(S2) 출연진 영역에서, `PerformanceArtist` 관계를 통해 매칭된 `Artist`를 칩(chip)으로 표시한다.
- 매칭되지 않은 출연진은 기존대로 `prfcast` 원문 텍스트로 표시(원문 보존 원칙 유지).

### F7.2 칩 클릭 동작
- 아티스트 칩 클릭 시 동일 아티스트의 다른 공연 목록을 조회한다.
- 아티스트 칩 클릭 시 목록 페이지로 이동하며 아티스트 필터를 적용한다: `/?artist=<Artist.id>`. F8(아티스트 필터)과 연계.

### F7.3 범위 제한
- **팔로우는 v3 범위 밖** — 아티스트 팔로우/알림은 후속 버전.
- Artist 엔티티의 하이브리드 수집(prfcast 추출 + MusicBrainz 매칭 + 수동 보정) 파이프라인은 P2 데이터 토대에서 구축.

## F8. 아티스트 필터 (v4 P3) · QA 티어: Deep

> 목록 페이지(S1)에서 아티스트를 선택하여 해당 아티스트 출연 공연만 필터링한다. 자체 Artist DB(data-model.md §3.5, §5.5) 기반.

### F8.1 입력 동작
- 아티스트 자동완성 입력(Autocomplete)을 제공한다.
  - 사용자가 2자 이상 입력하면 `artists` 테이블에서 `name`/`aliases` 기준으로 후보를 조회한다.
  - 후보 선택 시 해당 `Artist.id`를 필터에 적용한다.
- **단일 선택(v4):** 한 번에 하나의 아티스트만 필터링 가능. 다중 아티스트 필터는 후속 버전.

### F8.2 기존 필터와의 결합 규칙 (AND)
- **모든 필터와 AND 결합.** 아티스트 + 기간(F2.1/D1) + 지역(F2.2/D4) + 장르(F2.3) + 공연장(F2.4) + 검색어(F5) + 정렬(F4)이 동시에 적용된다.
- 아티스트가 선택되면 `performance_artists` 관계를 통해 해당 아티스트의 `mt20id` 집합을 구하고, KOPIS/BFF 결과와 교차(intersection)하여 필터링한다.

### F8.3 아티스트 미선택 시
- **기존 동작 불변.** 아티스트가 선택되지 않으면 아티스트 필터가 적용되지 않으며, 기존 필터 조합 결과가 그대로 표시된다.
- 디폴트 = 미선택 (D1 보강).

### F8.4 URL 동기화
- 아티스트 선택 시 쿼리스트링 `?artist=<Artist.id>`에 반영 (공유/북마크 가능, F2.5와 일관).
- 아티스트 미선택 시 `artist` 파라미터 생략 (디폴트 생략 규칙, data-model.md §8.2).
- 뒤로가기 복원 대상(D8)에 포함.

### F8.5 필터 초기화
- "필터 초기화"(F2.5) 시 아티스트 선택도 함께 해제한다.

### F8.6 데이터 소스 / 제약
- **데이터 소스:** 자체 Artist DB (`artists` + `performance_artists` 테이블, data-model.md §5.5).
- **누락 가능성:** prfcast 추출 파이프라인이 미처리한 공연은 아티스트 필터 결과에서 누락될 수 있다.
  - 목록 상단에 안내 문구 표시: "아티스트 매칭이 완료되지 않은 공연은 결과에 포함되지 않을 수 있습니다."
  - 안내 문구는 아티스트 필터가 활성화된 경우에만 표시한다.

### F8.7 칩 클릭 연계 (F7.2)
- 상세 페이지(S2)에서 아티스트 칩 클릭 시 목록 페이지(`/?artist=<Artist.id>`)로 이동하여 F8 아티스트 필터가 자동 적용된다.

## F9. 인증/계정 (v5 P4) · QA 티어: Deep

> 소셜 OAuth(카카오 + 구글)를 통한 로그인/로그아웃. ② Following, ④ My Record 등 사용자 기능의 공통 토대.

### F9.1 소셜 로그인
- 카카오(Kakao) OAuth 2.0과 구글(Google) OAuth 2.0을 지원한다.
- 로그인 페이지(S3)에서 카카오/구글 버튼 중 하나를 선택하여 로그인한다.
- 최초 로그인 시 간단한 약관/개인정보처리방침 동의를 받는다.
- 로그인 완료 후 원래 페이지로 redirect한다(callbackUrl).

### F9.2 세션/상태
- 로그인 후 세션이 유지된다(JWT 기반, Auth.js 관리).
- 헤더(S1 상단바)에 로그인 상태가 반영된다: 비로그인 시 "로그인" 버튼, 로그인 시 프로필 아바타+이름.
- 프로필 클릭 시 드롭다운: 로그아웃 버튼.

### F9.3 비로그인 사용자
- **기존 기능 완전 보장.** 비로그인 사용자도 모든 탐색 기능(F1~F8)을 사용할 수 있다.
- 인증은 v5에서 **선택적(optional)**. 인증 필수 기능은 v5 범위에 없음(Following/My Record는 후속).

### F9.4 로그아웃
- 로그아웃 시 세션 종료. 목록 페이지(S1)로 redirect. 필터 상태는 URL 기반이므로 유지.

### F9.5 User 엔티티
- User 데이터는 자체 SQLite DB에 영속 저장(비저장 원칙 완화, Venue/Artist 선례 확장).
- 저장 필드: id, email, name, image(프로필 사진 URL), provider(kakao/google), createdAt.
- Account/Session 테이블은 Auth.js adapter 요구사항에 따라 구성.
